# Your Toolbelt Map — How the Program Maps to Tools You Can Use

**AI Mentor Circle — Reference Guide**

Everything you do in this program is a real step in building a product. This map lines up each workflow with what you're really doing there, the artifact you walk away with, and the reusable **tool** the move becomes.

This class emphasizes **define** before you build, initial **build** before you design, and then **evaluate** and refine. Building before the deep design work is deliberate — a working first version shows you what actually needs designing. 

Each session has learnings you can capture in a reusable workflow or **tool**, a move you can run again on any future project.


| Session focus                          | Session artifacts                                                                                         | What you're really doing                                                                                                                                                                                                                                                            | Takeaway tool                |
| -------------------------------------- | --------------------------------------------------------------------------------------------------------- | ----------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- | ---------------------------- |
| **orientation** (S1)                   | `docs/reports/participant_profile.md` · `CLAUDE.md`                                                       | **Set up** — define the **persona** (who the user is and what they do) and define the product **outcomes** (the user's goals and what success looks like). It's what configures the AI to your product.                                                                             | *Set the Context*            |
| **ideation** (S2)                      | `docs/problem_definition.md` · `prompts/testing_prompt.md`                                                | **Define** — turn the loose idea into a sharp, scoped problem, then name and test the assumptions most likely to sink it before you build anything.                                                                                                                                 | *Problem Framing*            |
| **user_research** (S3)                 | `docs/user_research_plan.md` · `docs/user_research_findings.md` · `prompts/interview_analysis_prompt.txt` | **Validate** — turn messy user conversations into evidence-backed insights, separating what people actually said from what you hoped they'd say.                                                                                                                                    | *Interview Synthesis*        |
| **evaluation_dataset / spike** (S2–S3) | `docs/evaluation_design_report.md` · `data/evaluations_data.csv` · `docs/research/spike/`                 | **Prototype** — prove or kill the idea as cheaply as possible: simulate the flow in conversation or simplest prototype to surface problems for free, then build the thinnest spike and test it against real cases.                                                                  | *Simplest Prototype*         |
| **implementation** (S4)                | `docs/implementation_design.md` · `docs/specs/poc_specs.md` → built POC in `src/`                         | **Build** — sequence the build by risk: make the one slice that tests your biggest unknown first so you learn before you invest, then build out the rest. The same instinct as prototyping your shakiest assumption first.                                                          | *Prioritize by Risk*         |
| **context_architecture** (S5)          | `docs/context_architecture_design.md`                                                                     | **Design (behavior)** — watch where quality breaks in the running build, then decide how it should act, what it must never do, and how it handles uncertainty, and write those rules into the system itself. This is design without a screen: you're shaping behavior, not layouts. | *Behavior Spec*              |
| **user_experience** (S5)               | `docs/specs/user_experience_spec.md`                                                                      | **Design (experience)** — design the experience around a system that acts on its own: map what the user provides, sees, controls, and approves, then polish the interface. Familiar UX, new center of gravity.                                                                      | *Experience Spec*            |
| **evaluating_for_scale** (S6)          | `docs/evaluating_for_scale_design.md`                                                                     | **Evaluate** — decide what "good" means and measure against it — setting and defending a quality bar.                                                                                                                                                                               | *Evaluation and Improvement* |
| **first_deployment** (S6)              | `docs/deployment_design.md` · `docs/specs/deployment_specs.md`                                            | **Refine** — put it in front of real testers and learn from how they actually use it.                                                                                                                                                                                              | *Pilot With Users*           |
| **demoing_your_agent** (S7)            | `docs/demo_plans.md`                                                                                      | **Present** — turn what you built into a focused demo that lands one clear win.                                                                                                                                                                                                              | *Product Demo*               |


Each row — your learned **tool** — is a move you can turn into a reusable **workflow**. Practice by making a **workflow card** for any stage or move. The template and two worked examples live in `workflow_cards.md` (*Problem Framing*, *Interview Synthesis*); *Prioritize by Risk* is the one we extract together in Session 4, and you author the rest in Sessions 5 and 6.
